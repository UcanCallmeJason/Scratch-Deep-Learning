< 관계형 데이터베이스 모델링 요약 >

1. 모델링의 전체적인 순서
	
	업무파악 -> 개념적 데이터 모델링 -> 논리적 데이터 모델링 -> 물리적 데이터 모델링의
	
	- 업무파악 : 의뢰인과 잘 협력해서 의뢰인이 어떤 목적을 가지고 있는지 정확히 파악하는 것.
	- 개념적 데이터 모델링 : 해당 업무에 어떤 개념들이 서로 어떻게 상호 작용하는지 알아보는 것. (결과물 : ERD)
		# ERD는 정보, 정보 그룹, 정보 그룹간의 관계를 담고있다.
		# ERD를 만들고나면 논리적 데이터 모델링은 수월하다.
		# 관계형 데이터베이스는 개념들을 뽑아낼 때 내포관계를 허용하지 않는다.
	
	- 논리적 데이터 모델링 : 의뢰인과 생각했던 개념들과 관계들을 표로써 표현해보는 것.
	- 물리적 데이터 모델링 : 어떠한 데이터 베이스 제품을 쓸 것인지(ex MySQL) 결정한뒤, 제품을 이용하여 논리적으로 데이터를
					  표현한 표를 코드화하는 작업.
					  
2. Cardinality & Optionality

	Cardinality(차수)를 예를 들어 설명한다면, 
	담임 table과 학급 table이 있을 때, 담임 한명이 하나의 학급을 맡을 때, 하나의 학급이 한명의 담임을 부여 받을 때
		- 1 : 1 관계
	저자 table과 글 table이 있을 때, 저자는 글을 여러개 쓸 수 있고, 글은 저자를 한명만 가지고 있을 때
		- 1 : N 관계 ( 저자 : 글(N))
	저자 table과 글 table이 있을 때, 저자는 글을 여러개 쓸 수 있고, 글도 저자를 M명 가지고 있을 때 (책 한권을 쓰더라도 공동저자가 있을 수 있기 때문)
		- M : N 관계 ( 저자(M) : 글(N))

	-> ERD에서 표현할 때는 다수가 존재하는 테이블 쪽에 삼발이 표시를 해준다. 

	Optionality는 릴레이션을 표현할 때 말그대로 Option 같은 존재다.
	저자가 있어야 글이 존재하지만, 저자가 꼭 글을 쓰진 않기 때문에 저자에게 글은 있어도 그만 없어도 그만 (option) 같은 존재다.
	
	-> ERD에서 표현할 때는 선상에서 옵션인 쪽에 O를 그려넣어준다. 

3. 논리적 데이터 모델링
	ERD에서 표현된 것을 관계형 데이터 베이스에 맞게 표처럼 전환하는 것.
	
	이 작업에서 데이터 표현은 다음과 같이 변환된다.
	Entity -> Table
	Attribute -> Column
	Relation -> PK, FK
	 
	
	KEY 종류
	
	- 후보키 : 릴레이션에서 튜플을 유일하게 구별하는 속성, 속성집합으로 기본키로 사용가능 함.
	- 기본키 : 후보키 중에서 선택받은 Main Key다. 한 릴레이션에서 특정 튜플을 유일하게 구별할 수 있는 속성이다.
			NULL값을 가질 수 없으며 Unique 해야한다.
	- 대체키 : 후보키가 둘 이상일 때 기본키를 제외한 나머지 후보키들을 말한다.
	- 슈퍼키 : 한 릴레이션 내에 있는 속성들의 집합으로 구성된 키로써 릴레이션을 구성한 튜플들 중 슈퍼키로 구성된 속성의 집합과
			동일하지 않다.
	- 외래키 : 다른 릴레이션의 기본키를 참조하는 속성 또는 속성 집합으로, RDB에서 매우 중요함.
	
4. 키 부여 방법 (카디널리티에 대한 이해가 있다면 쉽다.)

	- 1:1 관계	: 누가 누구에게 의존하는가를 따져본다.
	- 1:N 관계	: 1쪽에 PK, N쪽에 FK.
	- N:M 관계	: 새로운 Mapping Table을 만든다. 양 쪽 테이블에서 고유한 식별자를 넣어만든 테이블을 참조해서 접근한다.
